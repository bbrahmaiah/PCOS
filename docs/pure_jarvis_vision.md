## PCOS Architecture Doctrine

The target is a Personal Cognitive Operating System, not a voice chatbot.
The project is organized around three layers:

```text
Layer 1 - Core, never casually rewritten
Identity
Memory
Reasoning
Safety
Mission Context
Event Bus
State Management
Tool Framework

Layer 2 - Capabilities, swappable over time
Speech
Vision
Research
Engineering
Simulation
Communication
Automation
Security
Health

Layer 3 - Intelligence Growth, governed by the core
Learning
Adaptation
Prediction
Habits
Preferences
Self-Optimization
```

Every live request must follow the protected flow:

```text
User
-> Voice
-> Perception
-> Mission Context
-> Memory
-> Reasoning
-> Planning
-> Safety
-> Execution
-> Feedback
-> Memory Update
```

Nothing important should bypass mission context, memory, safety, or governed
memory update. Models, STT engines, TTS engines, vision providers, and tool
adapters can be replaced, but the protected core must stay stable and
observable.

JARVIS may improve prompts, tune settings, learn habits, optimize latency, and
replace capability adapters. JARVIS may not rewrite its protected core, grant
itself unrestricted permissions, weaken safety policy, or perform destructive
or external actions outside the tool framework.

Continuity is the real product. Voice makes JARVIS accessible, speed makes it
usable, but memory plus mission context makes it feel alive.
